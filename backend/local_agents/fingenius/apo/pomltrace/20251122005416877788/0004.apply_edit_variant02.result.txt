### CRITIQUE

**Critique of the Prompt**

**1. Structural Issues:**  
- **Missing goal:** The prompt lacks a clear, one-line success definition (e.g., “Produce a final conclusion via terminate after data collection and analysis”).  
- **Contradictions:** The rule “直接执行下一个流程” conflicts with the tool restrictions—how does the model “execute” a stage without using tools if required?  
- **No stop condition:** Only the “综合结论阶段” explicitly ends the task; other stages lack completion criteria.

**2. Instruction Quality:**  
- **Vague verbs:** “判断哪些流程已经执行完毕” is ambiguous—no method is given to track state or progress.  
- **Mixed abstraction:** High-level workflow rules are mixed with implementation examples, creating confusion.  
- **Overlapping scope:** Repeated emphasis on not repeating stages appears in both the intro and “重要提示.”

**3. Control and Behavior:**  
- **Unclear uncertainty handling:** No guidance on what to do if the current stage is unknown or if data is incomplete.  
- **Verbosity confusion:** Example outputs demand brevity, but the prompt itself is verbose and repetitive.

**4. Input and Output Specification:**  
- **No input defaults:** The prompt assumes “当前状态” is always clear and contains needed context.  
- **Output schema missing:** Aside from example snippets, no consistent response format is enforced across stages.

**5. Scope and Safety:**  
- **No error handling:** No instructions for invalid states, like being in an undefined stage.  
- **User authority ambiguity:** The model must infer stage transitions autonomously, risking incorrect tool use.

**6. Efficiency and Maintainability:**  
- **Redundancy:** Multiple reminders not to repeat stages.  
- **Hidden assumptions:** Assumes the model can track “already executed” processes without a defined mechanism.

**Summary:**  
The prompt is logically inconsistent, lacks state-tracking clarity, and omits error handling. Restructure with explicit stage-transition logic, state variables, and consistent completion rules for each phase.
